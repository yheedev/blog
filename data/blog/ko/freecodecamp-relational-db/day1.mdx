# 1. 코드 에디터와 IDE의 차이점

코드 에디터와 IDE는 다른 개념이다. 코드 에디터는 코드 파일을 편집할 수 있는 모든 애플리케이션을 뜻한다. IDE(통합 개발 환경)은 코드를 편집하는 동시에 컴파일, 실행, 디버깅을 할 수 있는 완전한 애플리케이션이다. 코드 에디터는 주로 '코드 텍스트 자체를 편집하는 것'에 집중하고, IDE는 코드 관리를 위한 여러 개발 도구들을 함게 제공한다.

유명한 코드 에디터로는 Visual Studio Code, Sublime Text, Notepad++가 있고, 유명한 IDE로 Visual Studio, , Xcode, Android Studio가 있다.

위에 소개한 것들은 대부분 컴퓨터에 직접 설치해서 사용하는 로컬 프로그램이다. 하지만 클라우드 기반 에디터도 존재한다. 별도의 프로그램 설치 없이 웹 브라우저만으로 코드를 작성, 수정, 관리할 수 있는 온라인 도구이다. 유명한 클라우드 기반 에디터로 Replit, GitHub Codespaces, Ona가 있다.

어떤 도구는 VS Code처럼 확장성이 뛰어나서 여러 종류의 프로젝트와 언어를 지원할 수 있고, 또 어떤 도구는 특정 언어나 특정 프로젝트 유형에만 특화되어있다. 중요한 것은 현재 적합한 개발 도구를 선택해서 사용하는 것이다.

# 2. VS Code 생산성을 높여주는 유용한 단축키

모두 윈도우 기준이다

- `Shift + Alt + F` : 현재 열린 파일에 설정된 코드 포매터가 실행된다. JavaScript 파일의 경우, Prettier 도구가 사용된다.
- `Ctrl + Shift + F`: 워크 스페이스 전체 파일에서 특정 텍스트 검색
- `Ctrl + Shift + H`: 전체 파일에서 '찾기 및 바꾸기' 실행
- `Ctrl + Shift + K`: 해당 줄 바로 삭제
- `Ctrl + B`: 파일 목록이나 사이드바를 숨겨서 코드를 더 넓게 볼 수 있음
- `Ctrl + +`: 코드 텍스트 크기 확대
- `Ctrl + -`: 코드 텍스트 크기 축소
- `Ctrl + Alt + ↓`, `Ctrl + Alt + ↑`: 멀티 커서 기능을 통해 여러 줄을 동시에 수정할 수 있어서 반복 작업을 빠르게 처리할 수 있다.
- `Ctrl + Shift + P`: 명령 팔레트가 열리고, 원하는 기능을 검색해서 실행할 수 있다.

# 3. 터미널과 커맨드 라인의 차이점

커맨드 라인 (명령줄)은 텍스트 형태로 명령어command를 입력할 수 있는 기본 인터페이스이다. 주로 터미널 내에서 사용한다.
터미널은 기본적인 파일 읽기, 쓰기 작업과 시스템 수준의 명령어를 실행할 수 있도록 커맨드 라인 인터페이스를 제공하는 특별한 애플리케이션이다.

터미널 애뮬레이터는 기본적인 터미널 인터페이스를 감싸서 추가 기능이나 편의를 제공하는 애플리케이션이다. 일반적으로는 그냥 터미널이라고 부른다.

쉘shell은 커맨드 라인을 감싸고 있는 소프트웨어로, 사용자가 입력한 내용을 명령어로 해석하고, 그 결과를 반환하는 역할을 한다. 대표적인 쉘으로는 Bash, zsh, PowerShell이 있다.

예를 들어 터미널이 카페 매장이라면, 커맨드 라인은 주문을 입력하는 카운터에 가깝다. 그리고 셸은 주문을 해석해서 실제로 처리하는 직원이다.
터미널에서 `cd project` 같은 명령어를 입력하는 인터페이스가 커맨드 라인이다. 개발자가 `mkdir my-project` 같은 명령어를 입력할 때, 셸은 이걸 읽고 '새 폴더를 만들라는 뜻이구나' 하고 해석해서 OS에 전달한다. 즉, 셸은 명령어 해석, 실행, 결과 반환을 담당하며 커맨드 라인을 래핑해서 입력을 명령으로 해석하는 소프트웨어 프로그램이다.

윈도우에서는 터미널을 실행하기 위해서 기본적으로 PowerShell, Command Prompt(명령 프롬프트)라는 2개의 쉘 애플리케이션 중 하나를 사용한다. 이 둘은 각각 자신만의 애플리케이션을 갖고 있으며, 이름도 동일하다. PowerShell과 Command Prompt(cmd)는 Windows에서 제공하는 대표적인 셸이다. Windows Terminal은 이러한 셸을 탭 형태로 실행할 수 있는 터미널 애플리케이션이다.

윈도우 터미널이라는 터미널 에뮬레이터가 존재하며, 마이크로소프트 스토어에서 설치해서 파워쉘이나 커맨트 프롬프트처럼 실행할 수 있다. 윈도우 터미널은 파워쉘 쉘, 커맨드 프롬프트 쉘, WSL을 설정했을 경우 Linux 셸까지 모두 사용할 수 있다.

맥의 경우, 이름 그대로 Terminal 이라는 터미널 애플리케이션을 제공한다. macOS에서 iTerm2, Ghostty 같은 서드파티 터미널 애뮬리케이터를 사용할 수 있다.

리눅스의 경우, 기본 터미널과 실행 방법이 배포판과 데스크톱 환경에 따라 크게 달라진다. Linux는 서드파티 터미널 애뮬레이터 지원이 무척 풍부하다. Linux는 아마도 가장 다양한 터미널 애플리케이션을 제공하는 운영체제라고 볼 수 있다. Arch Linux에서는 kitty를 설치한 뒤, wofi 같은 애플리케이션 런처를 통해 실행할 수 있다. 대부분은 해당 배포판의 패키지 매니저를 통해 설치할 수 있다.

# 4. 생산성을 높일 수 있는 커맨드 라인 단축키

- 위쪽 방향키(↑): 이전에 실행했던 명령어 기록을 한 단계씩 거슬러 올라갈 수 있다.
- 아래쪽 방향키(↓): 명령어 기록을 다시 앞으로 이동할 때 사용한다.

- 많은 셸은 자동완성 기능을 제공해서 명령어 일부만 입력해도, 셸이 나머지 내용을 제안해주기도 한다. 이 때 `Tab` 키를 누르면 제안된 내용 중 하나를 선택해서 채워넣을 수 있다.
  - zsh에서는 최근에 사용한 명령어 기록을 기반으로 제안한다.
  - Powershell에서는 사용 가능한 명령어, 변수, 인자 등을 기반으로 제안한다.

- `cls`: 파워쉘에서 이 명령어를 사용해서 화면 전체를 지우고 새 프롬프트 상태로 만든다.
- `Ctrl + c`: 현재 실행 중인 명령에 인터럽트(SIGINT)를 보내 중단을 요청한다. 그러면 새로운 프롬프트가 나타나며 다른 작업을 계속 진행할 수 있게 된다.
  - 다만 파워쉘의 경우 Ctrl+c가 복사 기능으로 사용된다. 그래서 텍스트가 선택되지 않은 경우처럼 상황이 명확할 때만 프로세스 종료 기능으로 동작한다.
- `Ctrl+ z`: 파워쉘에는 없는 Unix 계열 터미널에서만 사용 가능한 명령어이다. 하나의 명령어를 백그라운드에서 실행한 상태에서 다른 작업을 동시에 해야할 때 해당 명령어를 통해 현재 실행 중인 프로세스를 백그라운드 작업으로 전환하고 다시 커맨드 라인으로 돌아올 수 있다.
  - 그리고 다시 그 작업으로 돌아가고 싶다면 `fg` 명령어를 입력해서 백그라운드 작업을 다시 가져올 수 있다.
- `!!`: 마지막으로 실행했던 명령어를 자동으로 다시 실행한다.

# 5. Bash의 개념과 기본적인 Bash 명령어

Bash는 “Bourne Again SHell”의 약자이며, Unix 계열 운영체제 환경에서 가장 널리 사용되는 셸(shell) 중 하나이다.

`pwd`는 현재 작업 디렉터리를 터미널에 출력한다. 작업 디렉터리란 현재 터미널이 위치하고 있는 폴더를 의미한다. 가끔 터미널이 내가 원하는 폴더가 아니라 다른 디렉터리를 가리키고 있을 수도 있다.

이 때 사용하는 명령어가 `cd` 이다. `cd`는 디렉터리를 변경(change directory)하는 명령어이다. `cd`에서는 `/`로 시작하는 절대 경로(absolute path) 사용, 접두사 없이 사용하는 상대 경로(relative path) 사용, 그리고 `..` 문법을 사용한 상위(parent) 디렉터리 이동이 가능하다.

`ls`는 현재 작업 디렉터리 안의 파일과 폴더 목록을 보여준다. 현재 디렉터리에 특정 파일이나 폴더가 존재하는지 확인할 때 매우 유용하다. `ls -a`를 사용하면 숨김 파일을 표시하고, `ls -l`을 사용하면 파일 권한 등 자세한 정보를 확인할 수 있다.

`cat`은 파일 내용을 한 번에 출력한다. `less`는 긴 파일을 페이지 단위로 스크롤하며 볼 수 있다.

원하는 파일이나 폴더가 아직 존재하지 않는다면 `mkdir`으로 새로운 디렉토리(폴더)를 생성하고, `touch`를 사용해서 새로운 파일을 생성할 수 있다. 두 명령어 모두 생성할 파일이나 폴더 이름을 인자로 받는다.

만약 `touch readme.md` 라는 새로운 파일을 생성 후, 파일 이름을 수정하고 싶다면 `mv` 명령어를 사용해서 파일 위치를 이동하거나 이름을 변경할 수 있다. `mv`는 기존 파일 이름과 새 파일 이름을 함께 입력받는다.

`rm` 명령어를 사용해서 파일을 삭제할 수 있다. `rm -r`은 디렉토리 삭제가 가능하고, `rm -f`는 삭제 확인 메시지를 무시하고 강제로 삭제한다.

`cp`는 파일이나 디렉토리를 새로운 위치로 복사(copy)할 때 사용한다. `mv`와 다르게 `cp`는 원본 파일을 삭제하지 않는다. 또 디렉토리를 복사하려면 `cp -r` 을 사용해야 한다.

`echo`는 Bash에서 `console.log` 같은 역할을 하며, 문자열을 입력받아서 터미널에 출력한다. 예를 들어 `echo "hello"` 처럼 사용할 수 있다. 터미널에서 입력해서 터미널에 다시 출력하는게 무의미하다고 느껴질 수 있지만, echo는 제어 기호와 함꼐 사용할 때 진가를 발휘한다.
`echo "Naomi was here." > readme.md`과 같이 사용하면, `>` 기호를 통해 문자열 내용을 파일에 저장할 수 있다. 이 경우 `readme.md` 파일이 새로 생성되거나 기존 내용이 덮어쓰기(overwrite)된다.

`>>` 기호를 사용하면 기존 파일 내용 뒤에 추가(append)할 수 있다.

이 외에도 `head`, `tail`, `chown`, `chmod` 등 많은 명령어가 존재하며 필요에 따라 학습하는 것이 권장된다. 거의 모든 명령어는 `man` 명령어를 통해 설명서나 도움말 페이지를 확인할 수 있다.

# 6. 명령어 옵션과 플래그

옵션과 플래그는 명령어의 동작 방식을 바꾸기 위해 추가로 전달하는 인자이다. 플래그는 옵션의 한 종류에 해당한다. 옵션과 플래그는 거의 같은 의미로 사용되지만, 일반적으로 플래그는 특정 기능을 켜거나 끄는(on/off) 역할을 하는 옵션을 의미한다.

예시

- `git status -s` 명령어에서 `-s`가 플래그이다.
- `ls --width=50` 명령어에서 `--width=50`이 옵션이다.
- `npm run dev -- --port 3000` 명령어에서 `--port 3000`이 옵션이다. `--port 3000` 앞의 `--`는 옵션 전달 구분자(separator)에 해당한다.

| 종류         | 특징                  | 예시                        |
| ------------ | --------------------- | --------------------------- |
| 플래그(flag) | 기능 켜기/끄기        | `-a`, `--help`, `-s`        |
| 옵션(option) | 값 전달해서 설정 변경 | `--port=3000`, `--width=50` |

옵션은 보통 `-` 또는 `--` 로 시작한다. 이렇게 하면 일반 인자와 옵션을 시각적으로 쉽게 구분할 수 있다.

`--`를 사용하는 긴 형식(long form) 문법은 다음과 같다.

- `ls --version`: 일반적인 명령 동작 대신 현재 프로그램 버전을 터미널에 출력
- `--help`: 해당 프로그램의 사용 방법, 옵션 설명이 출력된다.

`-`를 사용하는 짧은 형식(short form) 문법은 다음과 같다. `

- `ls -a`에서 `-a` 옵션은 숨김 파일까지 모두 표시한다. (.env, .gitignore 포함)

짧은 형식 문법의 장점은 여러 플래그를 한 번에 묶어서 사용할 수 있다는 점이다. 예를 들어 긴 형식에서는 `ls --all --human-readable --size`와 같이 여러 옵션을 쓰는 대신, 짧은 형식에서는 `ls -ahs` 처럼 한 번에 합쳐서 사용할 수 있다.

일부 옵션은 on, off 형식으로 사용하는게 아니라 특정 값을 함께 전달해야한다. 이 차이를 알아두지 않으면, 옵션 값(option value)을 일반 위치 인자(positional argument)로 잘못 전달하는 실수를 할 수 있다.

- 긴 형식에서는 `ls --width=50`와 같이 사용해서 보통 등호를 통해 옵션과 값을 연결한다.
- 짧은 형식에서는 `ls -w 50`와 같이 공백으로 값을 구분한다.

문법이 헷갈리면 언제든 `--help` 플래그를 사용해서 해당 명령어의 올바른 옵션 사용법을 확인할 수 있다.

# 7. 관계형 DB와 비관계형 DB의 차이점

관계형 데이터는 데이터를 행과 열으로 구성해서 테이블 형태로 정리해서 저장하는 DB이다.
행(row, 가로)는 개별 레코드를 나타낸다. 열은(coulmn, 세로)는 각 레코드를 설명하는 속성 또는 필드를 나타낸다.
예를 들어, 사용자 데이터를 저장한 테이블이 있다고 가정할 때, 각 행은 특정 사용자 1명을 의미할 수 있다. 그리고 이름, ID 같은 공통 속성은 열에 속한다.

관계형 DB의 핵심 특징 중 하나는 공통 속성을 통해 서로 다른 테이블을 연결(link)할 수 있다는 것이다. 이를 통해 데이터 간의 관계를 쉽게 표현하고 조회할 수 있다.

관계형 DB는 반드시 스키마를 필요로 한다. 스키마는 DB 전체 구조를 정의하는 설계도 같은 개념이다.
어떤 테이블이 있는지, 어떤 컬럼이 있는지, 각 컬럼의 데이터 타입이 무엇인지, 테이블 간의 관계는 어떻게 되는지, 어떤 제약 조건(constraint)이 있는지 등을 정의한다.

관계형 DB는 정해진 양식의 엑셀 같은 느낌이다. 대표적으로 PostgreSQL, MySQL이 있다. 모든 데이터는 정해진 규칙대로 저장하는 것이 필요하다.

예를 들어 아래에 '회원' 테이블 예시를 보자.

| id  | name | age | email      |
| --- | ---- | --- | ---------- |
| 1   | 희은 | 28  | a@test.com |
| 2   | 민수 | 31  | b@test.com |

여기서 모든 행(row)이 모든 컬럼(column) 구조를 가져야 한다.
그리고 name은 문자열, age는 숫자, email은 문자열과 같은 규칙(스케마)가 있다.

관계형 DB의 핵심은 테이블끼리 연결이 가능하다는 것이다. 아래에 '주문' 테이블 예시를 보자.

| order_id | user_id | product |
| -------- | ------- | ------- |
| 101      | 1       | 키보드  |
| 102      | 1       | 마우스  |

여기서 `user_id = 1`을 통해 '회원' 테이블에 있는 희은 데이터와 연결된다. 즉, 이 과정을 통해 이 주문이 누구 것인지를 정확히 확인할 수 있다.
그래서 데이터 정확성이 무척 중요한 은행, 쇼핑몰, 정부 시스템 등에 자주 사용된다.

관계형 DB의 모든 테이블은 기본 키(primary key)를 가진다. 기본 키는 각 행을 고유하게 식별할 수 있는 값으로, 여러 테이블의 데이터를 연결하는 핵심 요소이다.
또한 외래 키 (foreign key)를 통해 테이블 간의 관계를 연결할 수 있다.

<Spacer />

야생동물 보전 센터 기관에서 DB를 만든다고 가정해보자. 동물들의 일반 정보를 저장하는 animals 테이블이 만들어진다.
각 행은 동물 한 마리를 나타내고, 각 열은 종, 크기, 몸무게, 나이, 기타 특성을 나타낼 수 있을 것이다.
그리고 각 동물을 공유하게 식별하기 위해 보통 `id` 컬럼을 추가한다. 이 `id` 컬럼이 바로 해당 테이블의 기본 키(primary key)가 된다. 각 동물 ID는 다른 테이블과 데이터를 연결할 때 사용할 수 있다.

예를 들어 `species` 라는 테이블을 따로 만들어 서식지, 보전 상태 같은 종species 관련 정보를 저장할 수 있다.
그리고 animals 테이블의 각 동물은 species ID를 통해 이 테이블과 연결될 수 있을 것이다.

또한 동물들의 진료 기록을 저장하는 테이블도 만들 수 있을 것이다. 이를 통해 건강 상태, 치료 기록, 약물 정보 등을 추적할 수 있다.

이처럼 기본적인 스키마를 구성한 뒤, 여러 테이블을 animals 테이블과 연결하면 animals 테이블이 동물 관련 데이터의 중심 역할을 한다. 이 방식의 장점은 모든 정보를 하나의 테이블에 몰아넣지 않아도 된다는 점이다. 관련 데이터를 여러 개의 테이블에 나눠 저장하고, 관계(relationship)을 통해 연결할 수 있다.

이 관계를 잘 정리하면 단 하나의 쿼리만으로 원하는 데이터를 가져올 수 있다. 쿼리는 DB에 특정 데이터를 요청하는 작업을 의미한다.
예를 들어 건강 검진이 필요한 동물 찾기 같은 쿼리를 작성할 수 있을 것이다.

야생동물 보호 분야 뿐 아니라 관계형 DB는 의료, 건설, 비즈니스, 게임, 교육, 정부 기관, 전자 상거래, 소셜 미디어 등 매우 다양한 산업에서 사용된다.

관계형 DB의 대표적인 장점은 데이터 무결성과 일관성이다. 대규모 데이터셋을처리할 수 있기에 복잡한 실제 서비스에 적합하다.
또한 기본 키, 외래 키, 데이터 타입 등을 통해 무결성을 강제할 수 있다. 즉, 데이터의 일관성과 정확성을 유지하기 쉽다.

비관계형 데이터베이스(non-relational database)는 관계형 데이터베이스와 데이터를 저장하는 방식에서 차이점을 갖고 있다.
비관계형 DB는 NoSQL DB라고 불리며, 훨씬 더 유연한 구조를 가진다. 비관계형 DB는 데이터를 테이블과 관계 중심으로 저장하지 않는다. 대신 문서(document), 키-값(key-value), 그래프(graph), 컬럼 패밀리(column family) 등 다양한 방식으로 저장한다. 데이터 모델 자체도 더 자유롭다. 일부 비관계형 DB는 기본적인 스키마나 데이터 모델을 가질 수 있지만, 일반적으로 관계형 DB의 스키마만큼 엄격하지 않다.
필요에 따라 데이터 필드 추가, 수정, 제거하는 작업이 더 자유롭다.

비관계형 DB는 자유형 메모장에 가깝다. MongoDB는 문서(Document) 기반 NoSQL DB이고, Redis는 Key-Value 기반 NoSQL DB이다. 예를 들어 MongoDB는 데이터가 이렇게 생길 수 있다.

유저 A:

```JSON
{
  "name": "희은",
  "age": 28,
  "hobby": ["게임", "러닝"]
}
```

유저 B:

```JSON
{  "name": "민수",  "instagram": "@minsu"}
```

MongoDB는 관계형 DB보다 훨씬 유연한 구조를 허용한다. 같은 컬렉션 안에서도 문서마다 필드 구성이 달라질 수 있다.

데이터 구조가 자주 바뀌고, 빠르개 개발해야 하고, 유연성이 중요할 때 사용하면 좋다. JSON처럼 다루기가 편하다.
주로 채팅 앱, SNS, 로그 저장, 실시간 데이터 등에 자주 사용된다.
단점으로는 데이터 중복이 많아지고, 데이터 정합성 관리가 어렵다.

관계형 DB와 비관계형 DB 중 무엇을 선택할지는 여러 요소에 따라 달라진다. 데이터의 성격, 애플리케이션의 요구사항, 확장 방식, 데이터 구조 변화 가능성 등을 고려해야 한다.
