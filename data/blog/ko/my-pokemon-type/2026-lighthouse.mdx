> Render Blocking CSS?

브라우저는 CSS를 `<head>`에서 발견하면 다운로드 완료 전까지 화면을 아예 그리지 않는다. 이게 render blocking에 해당한다.

> 브라우저 요청 -> HTML 수신 -> CSS 발견 -> CSS 다운로드 기다리는 동안 흰 화면 → CSS 완료 → 렌더링 시작

render blocking이 왜 발생하냐면 Next.js가 빌드할 때 CSS를 자동으로 청크로 분리해서 `<link rel="stylesheet">`로 스타일을 주입하기 때문이다. 이 태그 자체가 브라우저 스펙상 render-blocking에 해당한다. CSS가 먼저 막고, 그 CSS 안에 폰트 참조가 있어서 폰트가 추가로 막는 구조이다.

폰트 blocking은 1. 에서 해결했고, CSS blocking은 next.js가 자동 생성하는거라 직접 제어가 어렵다. 96 -> 100점 상승에 있어서 큰 영향은 없을 것이라는 판단이 들어 그냥 두기로 했다.

이 이슈는 다크모드 초기화 순서와 관련된 문제였다. 현재 다크모드 초기화 순서는 아래와 같다.

1. 페이지 렌더링 (body에 아무 클래스 없음)
2. JS 로드 → localStorage 읽기
3. body에 'light' 클래스 추가 → 레이아웃 시프트

해결하기 위해 `<head>`에 인라인 스크립트를 넣어서 JS가 로드되기 전에 동기적으로 클래스를 적용했다.

localStorage 키는 `mypkmn-theme`, zustand persist 구조는 `{ state: { theme } }`입니다. `<head>`에 인라인 스크립트를 추가했다.
이제 HTML 파싱 중 `<head>` 인라인 스크립트가 동기적으로 실행되어, React가 hydrate되기 전에 `body`에 `light` 또는 `dark` 클래스가 적용된다. JS 로드 후 클래스가 추가되는 타이밍 차이가 사라져 CLS가 제거되고 LCP가 개선된다.

```tsx
// 기존 src/app/layout.tsx

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ko" className={`${noto.variable} ${helios.variable} ${noto.className}`}>
      <body>
        <QueryProvider>
          <TooltipProvider delayDuration={300}>
            <ServiceWorkerRegister />
            <main>{children}</main>
            <Toaster duration={1000} position="top-center" />
          </TooltipProvider>
        </QueryProvider>
      </body>
    </html>
  )
}
```

```tsx
// src/app/layout.tsx

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
```

# 라이트하우스 측정 환경을 시크릿 모드로 변경

여기까지 해결한 뒤 배포하고 다시 라이트하우스 점수를 측정했는데 성능 96점에서 점수 자체에 큰 상승이 없었다.
이 때 시크릿 탭에서 측정을 해보니 99점이 나왔다.

![라이트 하우스 두번째 측정](/static/images/my-pokemon-type/lighthouse2.png)

아마 기존 탭에서는 브라우저 확장 프로그램, 쿠키, 로컬 스토리지 등 여러가지 요소들이 라이트하우스 측정에 영향을 주고 있었던 것으로 추측된다. 라이트하우스 점수를 측정할 때에는 기본 탭과 시크릿 탭 중 어디서 측정하는게 표준인지 찾아보았다. 그리고 시크릿 모드가 표준이라는 내용을 구글 라이트하우스 공식 문서에서 찾을 수 있었다.

라이트하우스 결과는 확장 프로그램, 쿠키, 로컬 스토리지 영향을 받기 때문에 시크릿 모드에서 실행되는 것이 권장된다.

# 📍 PERFORMANCE 5차 개선: 99점 → 100점

## 1. 레거시 JS 14KiB 제거 (실패)

폴리필이 Next.js 자체 번들(`next` 패키지)에 포함되어 있다. `i?.()` 패턴으로 `if (!X.prototype.at)` 형태로 조건부 주입되는데, 이건 Next.js 프레임워크가 번들에 직접 포함하는 것이라 `browserslist`나 SWC 설정으로 제거할 수 없다. `TURBOPACK` 번들은 `.env.local`의 `NEXT_FORCE_WEBPACK=true`가 로컬에서만 webpack을 강제하고, Vercel 배포 시에는 Turbopack이 동작해서 Turbopack 자체의 polyfill이 포함된다.

Turbopack의 polyfill은 현재 외부에서 제거할 방법이 없다. 이건 Next.js/Turbopack의 내부 동작이라 `browserslist`나 `experimental` 설정으로 우회할 수 없다.

## 2. CLS 0.074 (body.light)

`body.light`에서 레이아웃 변경이 발생하는데, 두 가지 원인이 추정된다.

1. 원인 A: layout-client.tsx의 `ready` 상태가 `false`일 때 스켈레톤을 보여주다가 실제 콘텐츠로 교체될 때 레이아웃이 흔들림
2. 원인 B: 폰트 로딩 (`display: 'optional'`은 swap period가 없어 스왑은 발생하지 않지만, block period ~100ms 내에 로드되지 못하면 폴백 폰트로 고정되어 레이아웃에 영향을 줄 수 있음)

스켈레톤이 실제 콘텐츠와 높이가 다르면 CLS를 유발한다. 스켈레톤 크기를 실제 레이아웃에 맞게 조정하거나, `ready` 조건 자체를 제거하는 방향으로 개선 가능했다.

CLS 원인을 정확히 파악한 결과, `layout-client.tsx`가 `useEffect`에서 i18n을 초기화하고 `ready=true`가 될 때까지 스켈레톤을 보여주는데, 스켈레톤 → 실제 콘텐츠로 교체될 때 레이아웃이 흔들린다. 그런데 `/src/lib/i18n.ts`를 보면 번역 파일이 이미 JS 번들에 포함되어 있어서 `initI18n`은 사실 동기 함수이다. `useEffect`를 기다릴 필요가 없다.

그래서 `ready` 상태 + 스켈레톤 완전 제거하고, `initI18n(lang)`을 렌더 중 동기 호출했다. 번역 파일이 번들에 이미 포함되어 있으므로 가능하다. `setLanguage`, `saveLang`, `initTheme`은 side effect라 `useEffect`에 유지했다.

CLS는 스켈레톤 → 실제 콘텐츠 교체가 없어지므로 0.074에서 더 줄어들고, LCP는 렌더링 지연 650ms 제거되는 것을 기대할 수 있었다. (스켈레톤 대기 없이 바로 실제 콘텐츠 렌더)
