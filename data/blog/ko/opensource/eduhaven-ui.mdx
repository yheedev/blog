> [PR 링크](https://github.com/EduHaven/EduHaven/pull/394#issue-3334248947)

https://goodfirstissues.com/ 에서 Eduhaven의 저장소의 [이슈](https://github.com/EduHaven/EduHaven/issues/362#event-19196341022)를 발견했다. 할당 요청 코멘트를 달고 할당을 받아서 작업을 시작했다.

<Spacer />

해당 이슈에서는 기존에는 로그인 후의 메인 페이지의 카드 컨텐츠를 렌더링 하기 위해 로딩 중일 때 유저에게 서클 형태가 표시 되었는데, 스켈레톤으로 변경하는게 필요하다고 했다.

이 이슈를 해결하면서 레이아웃 시프트가 없어야 한다는 조건이 있었다. 레이아웃 시프트는 요소가 갑자기 움직이거나 크기가 달라져서 사용자가 보던 위치가 바뀌는 현상을 뜻했다. 그러니까, 스켈레톤의 크기나 위치가 실제 나타나는 데이터의 크기와 위치와 거의 동일하게 구현되어야 한다는 뜻이었다.

EduHaven은 20개의 테마를 지원하고 있었는데, 크게 라이트테마와 다크테마로 나눠져 있었다. 개별적인 테마에 맞는 20개의 스켈레톤을 컬러 별로 구현하기 보다는 범용적으로 사용하기에 적절한 스켈레톤 UI를 라이트, 다크 테마의 스타일만 염두에 두고 만들기로 했다.

# 테스트 계정 만들기 위해서 MongoDB 서버 세팅

https://www.mongodb.com/ko-kr

구글 계정으로 가입 -> deploy your cluster에서 free 선택, configurations에서 provider는 아무거나 골라도 무방하다. region만 서울 고르면 됨. create cluster 버튼 누르고 기다리면 된다.

이후 좌측 메뉴의 Database Access 클릭, Add New Database User 클릭
Add New Database User 페이지 진입 후:

- Authentication Method에서 Password 클릭
- Password/Authentication에서 테스트 계정으로 사용할 아이디, 비밀번호 입력
- Built-in Role에서 Read and write to any database 선택
  add user 버튼 클릭 후 저장

다시 좌측 메뉴의 Network Access 클릭, Add IP Address 클릭

- Access List Entry 값으로 0.0.0.0/0 입력 후 Confirm 버튼 클릭해서 저장한다.
  짧은 개발, 테스트 용도로 괜찮지만 보안에는 취약하다. 실제 운영이나 장기간 유지가 필요하면 개인 IP로 한정하는게 좋지만 빨리 세팅하는게 목적이니 생략했다. 작업이 끝난 뒤 빠르게 삭제했다.

좌측 메뉴의 Cluster 클릭,

- connect 버튼 클릭
- 이후 Drivers 선택
- 3. Add your connection string into your application code에 있는 mongodb+srv...로 시작하는 코드 복사 해두기
- done 버튼 눌러서 저장

# 스켈레톤 UI 추가

MongoDB 세팅을 기다리며 스켈레톤 추가 작업 계획을 다음과 같이 세웠었다.

1. 기존에 다른 곳에서 사용 중인 스켈레톤 컴포넌트가 있는지 확인해서 공통 컴포넌트를 사용하자.
2. 공통 스켈레톤 컴포넌트가 없다면 shadcn 이나 radix 와 같은 UI 라이브러리를 사용하는지 여부를 확인한 뒤에, 해당 라이브러리에서 스켈레톤 UI를 다운로드 받아서 사용하자.
3. 만약 사용 중인 UI 라이브러리가 없다면 Tailwind를 사용해서 스켈레톤 컴포넌트를 새롭게 생성하기로 했다.
4. page.tsx 혹은 어떤 클라이언트 컴포넌트에 스켈레톤 UI 추가해야할지 확인하자.

다른 파일을 둘러보니 다른 곳에서 사용 중인 스켈레톤 UI가 이미 있었다.

범용적으로 사용할만한 스켈레톤 컴포넌트를 생성할지 말지 고민을 했다. 그렇지만 나는 당시에 현재 이슈를 해결하고 더이상 이 프로젝트에 참여할 계획이 없었기 때문에 범용적으로 사용될 컴포넌트를 생성하기에는 컨트리뷰터와 추가적인 대화가 필요하겠다는 생각이 들었다. 아마 개인 프로젝트였다면 스켈레톤 컴포넌트를 따로 만들었을텐데, 오픈소스 기여 경험이 적다보니 내가 참여해도 되는 범위가 어디까지인지를 고민했던 것 같다.

다양한 테마에서 테스트를 거친 뒤 캡쳐를 해서 PR 템플릿에 첨부하는게 필요했다. 로딩이 너무 순식간에 이뤄졌기 때문에 테스트할 때는 아래 코드를 임의로 추가해서 캡쳐를 했다. 커밋 전에는 모두 지워두었다.

```
// TimeStats.jsx

const MIN_SKELETON_MS = import.meta.env.DEV ? 1000 : 0; // 개발: 1000ms, 프로덕션: 0ms
const delay = (ms) => new Promise((res) => setTimeout(res, ms));

const fetchUserStats = async (isRefresh = false) => {
  try {
    if (isRefresh) setRefreshing(true);
    setError(null);

    // 스켈레톤 최소 노출을 위해 요청과 delay를 동시에 수행
    const token = localStorage.getItem("token");
    const url = `${String(backendUrl).replace(/\/$/, "")}/user-stats`;
    const headers = token ? { Authorization: `Bearer ${token}` } : {};

    // 실제 요청과 인위적 지연을 병렬 실행
    const [response] = await Promise.all([
      axios.get(url, { headers }),
      delay(MIN_SKELETON_MS),
    ]);

    const stats = response.data;

    setStudyData({
      Today: `${stats.timePeriods.today} h`,
      "This week": `${stats.timePeriods.thisWeek} h`,
      "This month": `${stats.timePeriods.thisMonth} h`,
      "All time": `${stats.timePeriods.allTime} h`,
    });

    setUserStats({
      rank: stats.rank,
      totalUsers: stats.totalUsers,
      streak: stats.streak,
      level: stats.level,
    });

    setLoading(false);
    setRefreshing(false);
  } catch (err) {
    await delay(MIN_SKELETON_MS);
    setError("Failed to load statistics");
    setLoading(false);
    setRefreshing(false);
  }
};
```

# 컨트리뷰터와 스켈레톤 방식에 대한 논의

컨트리뷰터가 빠르게 콘멘트를 달아주었는데, 이런 내용이었다.

> 전반적으로 괜찮아 보이지만, 스켈레톤은 오직 textual data만 적용되어야 합니다.

내가 텍스트 컨텐츠가 아닌 드롭다운, 아이콘에 대한 스켈레톤을 적용했는데 이를 제외해달라는 요청을 하는 것 같았다. 하지만 내가 전체적으로 살펴본 전체 페이지에서는 분명 아이콘에 대한 스켈레톤이 적용되어 있었다.

나는 EduHaven의 배포 페이지 /stats 페이지에는 스켈레톤에서 정적 요소도 구현되어있는 것을 확인했고, 전반적인 일관성을 고려했을 때 아이콘에 대한 스켈레톤을 구현하는 것이 맞다고 느꼈다고 코멘트를 달았다.

그리고 컨트리뷰터로부터 확실히 일관성이 없다는 것을 인정했고, PR 병합 후 추후 스켈레톤 동작을 일관적으로 조정하는 작업을 진행하겠다는 답변을 받고 이슈는 병합이 되었다.

# 기여가 끝난 뒤 후기

ComposeFlow 이슈를 해결할 때 UI 변경 작업보다 Android Studio 세팅에 많은 시간을 썼던 것처럼, 이번에도 UI 작업 자체 보다는 테스트용 계정을 만들고, MongoDB 세팅을 하는 데에 더 많은 시간이 들었다. 테스트용 계정이 있었으면 좋았겠다는 생각이 드는데, 꼭 테스트용 계정이 없더라도 로그인 전용 페이지에 진입할 수 있는 다른 더 편한 방법이 있었지 않을까 싶다. 일단 로컬스토리지나 세션에 임의로 데이터를 작성해두는 방법은 통하지 않았기 때문에... 일단 테스트용 계정이 있는 오픈소스를 선택했다면 좀 더 편했겠다는 생각이 들었다. 하지만 보안 때문에 테스트용 계정이 없다는 점은 이해가 갔다.

컨트리뷰터와 어떤 컴포넌트에 스켈레톤 UI가 필요한지에 대해 논의를 하면서 스켈레톤의 존재 의의를 한 번 생각해볼 수 있었다. 이번에는 전체적인 일관성을 논리로 들었지만, 그 외에 스켈레톤의 의의를 고민했을 때 유의미한 특징을 집어서 논리로 들었다면 좋았을 것 같다.

너무 세세한 스켈레톤의 단점은 무엇일까? 난 개인적으로 세세한 디테일을 구현하는 것을 좋아하는 편이라 솔찍히 나의 개인적 선호가 들어갔음을 인정해야할 것 같다. 앞으로 성능 상이나 UX에 있어서 텍스트 컨텐츠 외의 스켈레톤을 구현했을 때 어떤 단점이 있을까 좀 고민을 하면서 만들어보는게 필요할 것 같다.
